GTO Duel — Mobile Redesign Master Spec

A single, complete specification for the GTO Duel mobile redesign. It covers the touch / affordance overhaul, the compression of the whole hand workflow, the rebuilt results screen, and the view toggle. Every value uses the app's existing CSS custom properties and class names. The goal throughout: make the app fast and obviously interactive on a phone, fit each stage of the loop in one viewport, and keep the dark premium poker aesthetic intact.

This document is self-contained and is the intended hand-off artifact. Visual mockups for every section are listed in Appendix B and live alongside this file.

1. Principles

These five ideas drive every decision below.

1.1 Affordance over decoration. The app's foundation is strong; the problem is that interactive elements are drawn like static containers. Spend the design system — elevation, shadow, fill, motion — on the things that are actually interactive.

1.2 Vertical real estate is the budget. On a stacked mobile layout the viewport is the constraint. Every element is measured in vertical pixels. Small savings, applied everywhere, are what keep the workflow compression-capable.

1.3 Every stage of the loop fits one viewport. Hand display, input, results, and the return to the next hand each resolve within roughly one ~700px screen, with no scroll required to reach the primary action.

1.4 Three engagement modes. Content on the results screen is one of three kinds, and each behaves differently:

  • Scan & react — glanceable, always visible, no interaction (verdict, crowd).
  • Read & absorb — study material, expanded inline on demand (the GTO reasoning).
  • Explore & do — an interactive tool, launched into a full surface (the equity tester).

1.5 Progressive disclosure, two gestures. What isn't in the always-visible spine is one of: an expand (inline accordion, for reading) or a launch (fullscreen, for tools). The disclosure gesture signals which: a chevron ▾ expands, an arrow ↗ launches.

2. Design system

2.1 Tokens

The app already defines a token system; keep it. Add the tokens in Appendix A. The one critical change: --border #243646 measures 1.30:1 against the panel background — below the 3:1 WCAG non-text-contrast minimum. It is reserved for dividers only. Anything interactive uses --border-interactive #3a5670.

2.2 Five-tier interaction system

Every interactive element belongs to exactly one tier.

Tier Name Used for Fill Border Depth States
0 Static labels, decorative tags, readouts transparent / faint tint none none none
1 Tap surface matrix cells, action-log rows --bg-elev-2 1px --border-tap inset bevel :active + :focus-visible
2 Secondary control Customize, Clear, playback, iteration select, view toggle --bg-elev-2 1px --border-interactive --shadow-1 :active + --accent-soft + :focus-visible
3 Primary action Run simulation, Next hand, move buttons --accent solid --accent accent glow :active + brightness + :focus-visible
4 Explore / open villain range cards, "Edit range" --bg-elev-2 + identity bar 1px --border-interactive --shadow-2 :active + glow + :focus-visible

2.3 The base patch — apply once, globally

The single highest-impact change. The app currently has zero :active rules, so a tap produces no confirmation, and :hover sticks on touch. This ~25-line block fixes press feedback, the 300ms tap delay, sticky hover, and keyboard focus across the whole app.

button, [role="button"], .rp-cell, .villain-range-card,
.spot-sum-action, .chip-action, .term-trigger{
  -webkit-tap-highlight-color: transparent;
  touch-action: manipulation;
  transition: transform .08s ease, filter .12s ease,
              background .12s ease, border-color .12s ease;
}
button:active, [role="button"]:active,
.rp-cell:active, .villain-range-card:active,
.spot-sum-action:active, .chip-action:active{
  transform: scale(.97); filter: brightness(1.06);
}
.rp-cell:active{ transform: scale(.88); }

button:focus-visible, [role="button"]:focus-visible,
.rp-cell:focus-visible, .villain-range-card:focus-visible,
.spot-sum-action:focus-visible{
  outline: 2px solid var(--accent); outline-offset: 2px;
}

/* move every existing :hover rule inside this block */
@media (hover: hover){ /* .rp-cell:hover { … }  button:hover { … } */ }

2.4 Shape rule — pill = button, box = everything else

  • Pill (fully rounded, border-radius: 999px) means a tappable action — and nothing else. Primary and secondary buttons may be pills.
  • Box (border-radius: 3–4px) is for tags, badges, status chips, count markers — every non-action label, including tags that are tappable for a definition. A tappable tag is still a tag, not a button.
  • Boxes are kept vertically tight: padding: 2px 6px, ~9px font. A box tag is ~17px tall vs a pill's ~23px — that 6px, repeated down a screen, is real budget (Principle 1.2).

3. Accessibility requirements

Ship these alongside the visual work, not after.

  • Focus. Every interactive element has a visible :focus-visible ring (2px --accent, outline-offset so it isn't clipped). Never outline: none without a replacement.
  • Semantics. Anything tappable is a <button>, or has role="button" + tabindex="0" + Enter/Space handlers — matrix cells, action-log rows, glossary terms, range cards included.
  • Accessible names. Icon-only and content-only controls carry an aria-label. Toggles expose aria-expanded or aria-pressed. Accessible names are never sourced from ::before / ::after generated content.
  • Target size. WCAG 2.2 SC 2.5.8 minimum is 24×24px; Apple HIG is 44. All controls clear 44 except the inline matrix — resolved by the fullscreen editor (§5).
  • Contrast. Interactive borders ≥3:1 against their background (--border-interactive clears it; --border does not, at 1.30:1). Control text ≥4.5:1.
  • Motion. Non-essential animation gated behind prefers-reduced-motion:
@media (prefers-reduced-motion: reduce){
  *{ animation-duration:.01ms !important; transition-duration:.01ms !important; }
}

Current: amber full border + faint tint — reads as a warning callout. Re-cast as an elevated explore card; amber moves to a 3px left identity bar (a full amber border reads as a warning). See §8.4 for the range-card structure on the results screen.

Per the shape rule (§2.4): decorative and interactive tags are small boxes; only buttons are pills.

  font-size:9px; font-weight:700;
  background:rgba(76,182,255,.10); border:1px solid rgba(76,182,255,.32);
  color:#9fd0ef; border-radius:3px; padding:2px 6px;
}
.tag.tappable{ cursor:pointer; }      /* e.g. glossary concept tags — still a box */

4.6 Inline glossary terms — .term-trigger

Keep the blue dotted-underline convention. Change cursor:help → cursor:pointer, widen the hit area with padding:2px 1px, ensure each is a real <button> with an accessible name.

5. The hand matrix

The 13×13 matrix renders at 20×20px cells in the in-game equity tester and 24.6px in the standalone Equity Calculator (measured at a true 390px viewport) — far below the 44px touch minimum and the 24px WCAG minimum. Two requirements:

A. Make it look editable. Apply the Tier-1 cell styling in §4.3 — raised cells, visible borders, 5px radius, glow on selected cells. Add a header above the grid: a small EDITABLE box tag plus "Tap to toggle hands."

B. Make it tappable — fullscreen editor (primary). The inline matrix is a compact preview; editing happens in a fullscreen surface (the equity tester, §8.5). A fullscreen sheet has no competing page scroll, so the grid can use the full screen width and cells reach a real touch target. Do not combine an inline horizontally-pannable matrix with drag-to-paint on a page-scrolling screen — page scroll, horizontal pan, and paint-select are three gestures that cannot share one surface. If a fullscreen surface is out of scope, the fallback is an inline overflow-x:auto matrix with grid-template-columns:repeat(13,minmax(34px,1fr)) supporting pan or drag-paint, never both.

6. The compressed workflow — four stages

The expanded hand view exceeds one viewport: the decision controls sit ~3 scrolls below the oval table, so when the table animates, the information needed to act is off-screen. The compressed workflow makes every stage fit one viewport. A view toggle (§7) switches between the expanded and compressed layouts.

6.1 Stage 1 — hand display

The oval 6-seat table collapses to a board runout strip (flop / turn / river grouped once, with a pot badge). A hero strip shows seat, hole cards, stack. The street-by-street log becomes a one-line timeline — one compact row per street: street tag + colour-coded H/V actions + running pot. The current decision point is highlighted. Board, hand, and full history in one viewport.

6.2 Stage 2 — input receive

Board and hand condense to a single strip at the top so the decision panel — move buttons, the confidence row (1–5), and the Lock in & see GTO button — all sit in one viewport. The input is never below the fold.

6.3 Stage 3 — reveal & results

The full results architecture; see §8.

6.4 Stage 4 — next hand

Next hand → is anchored at the bottom of the results screen, always reachable, and returns straight to Stage 1 — no transition screen, no scroll.

7. The view toggle (expanded ⇄ compact)

A control to switch a hand between the expanded (animated table) layout and the compressed layout.

Placement. A 32×32px icon button, absolutely positioned in the top-right corner of the scenario card, level with the "Scenario #NNN" label. Because it is absolutely positioned it adds 0px of layout height; because it sits in the card's dark header it is above the felt, clear of the table's animation layer. It is not placed on the felt (animation collision; would need backdrop-filter, costly on mobile Safari) and not in the playback row (at 390px that row has no spare space once the status label is present).

The control.

.view-toggle{
  position:absolute; top:11px; right:11px;
  width:32px; height:32px; border-radius:9px;
  background:var(--bg-elev-2); border:1px solid var(--border-interactive);
  color:#cfe6ff; cursor:pointer;
  display:flex; align-items:center; justify-content:center;
  transition:transform .08s ease, background .12s ease, border-color .12s ease;
}
.view-toggle:active{ transform:scale(.9); background:var(--accent-soft); }
.view-toggle:focus-visible{ outline:2px solid var(--accent); outline-offset:2px; }
.view-toggle[aria-pressed="true"]{ background:var(--accent); border-color:var(--accent);
  color:#061018; }

A real <button>, aria-label="Switch to compact view", aria-pressed toggling. The chevron icon points inward (collapse) in expanded view, outward (expand) when compact is active.

Suggesting it. The first time a hand view overflows the screen (scrollHeight > innerHeight), a one-time coach mark fades in pointing at the toggle: "This hand runs long — switch to a compact layout." It is an absolute overlay (0px layout cost), auto-dismisses on any tap or after ~6s, and is then stored as seen so it never repeats. No persistent badge or dot.

8. The results screen architecture

The results screen is built on three pillars — outcome / social, villain's range, and the explanation — with the GTO answer summarised at the top. It behaves as a hub: the spine is always visible; reading material expands inline; the equity tester launches fullscreen.

Section order, top to bottom:

  1. GTO summary card (§8.1)
  2. How everyone answered (§8.2)
  3. Villain's range (§8.4)
  4. Study the line (§8.6)
  5. Your take (§8.7)
  6. Next hand — anchored

8.1 GTO summary card

One compact card replaces what were three stacked elements — the lesson header, the verdict band, and the (previously buried) GTO reasoning. ~84px, three rows:

  • Row 1 — lesson + concept tags. The lesson is the screen's title, on one line (e.g. "Squeeze vs an open-and-caller"), followed by GTO concept tags as compact boxes (Capped range, Dead money, …). Tags are tappable to a glossary definition and double as the database's concept index. The header is results-only — showing the concept during the hand would hand the player the answer.
  • Row 2 — GTO strategy bar. A single horizontal bar segmented by the solver's frequency across the actions (e.g. Squeeze 76% / Call 18% / Fold 6%). The GTO-preferred segment is green; the player's pick segment is pink with a YOU pin. This is the high-level GTO view — the whole strategy at a glance — and the bar visually is the verdict.
  • Row 3 — verdict + EV + why. One line: the ✓/✗, the EV cost of any deviation (e.g. "≈1.4 BB"), and the core reason in a few words. The EV cost is the single most trainer-relevant figure — it quantifies the mistake.

The full reasoning paragraph is not here — it is one tap down in "Study the line" (§8.6).

8.2 How everyone answered (the crowd / social pillar)

The crowd database is the always-visible social spine — what every player sees on every hand. It is a list of answer rows, one per action, ordered by share. The rows are a hierarchy:

  • The player's chosen row is an elevated, distinctly-framed card — tinted fill, a left accent bar, a lifted shadow, a taller bar and a larger percentage, and a filled YOUR PICK marker with the player's avatar. It reads unmistakably as selected. A BLIND SPOT tag appears when the pick is off-line.
  • The GTO line row is its own flagged card (✓ GTO LINE, green-bordered).
  • Every other row collapses to a thin, dimmed line — present for completeness, out of the way.

Each row carries the avatars of the players who chose that action — see §8.3.

8.3 Comments — an indicator, not a section

There is no comment feed. A player who left a take has a green glow ring on their avatar in the crowd rows. The take text appears on hover (desktop) or tap (mobile) as a tooltip — discoverable without occupying the screen. (On touch, the tap target of a glowing avatar must not collide with other targets in its row.)

8.4 Villain's range (its own pillar / subsection)

Villain's range is one of the three results pillars, so it is a titled subsection — peer to "How everyone answered" — not an accordion row. It holds a list of range cards, one per sub-range the solver assigns villain (e.g. the capped flat-call range; the dominating offsuit aces). Each card is compact, ~52px:

  • a mini range-grid thumbnail — a ~46px non-interactive 13×13 heat grid showing the range shape at a glance;
  • the range name, a one-to-two-line description, and a combo count;
  • a launch affordance (↗).

The card is the content (the range read). Tapping it launches the fullscreen equity tester (§8.5) loaded with that range — the tool. Content stays in the section; the tool launches.

8.5 The equity tester — fullscreen tool

Launched from a villain range card. A full-screen surface: a back control, the hero hand and board, the loaded range chip, Clear / Customize (Tier-2 controls), the matrix (now with room — see §5), an iteration selector, Run simulation, and the equity result. Because it owns the whole screen there is no competing page scroll, so the matrix and drag-to-paint behave correctly.

8.6 Study the line — the explanation pillar

The full GTO reasoning (the read paragraph and the for/against breakdown of each option) is study material — read & absorb. It lives in an accordion row, Full GTO reasoning, that expands inline (chevron ▾). It is reference, reachable on demand, never in the way.

8.7 Your take — the player's public note

A section for the player to record their take on the scenario. The take is public — other players see it on the crowd breakdown (it is what lights the green glow in §8.3).

  • Entry point — one compact row on the results screen. Dashed when empty ("Add your take on this hand · Other players see it on the crowd breakdown"); solid with a green left edge when filled, showing the take truncated to two lines with an edit pencil.
  • Editor — tapping the row opens a bottom sheet with the textarea, a character hint, Cancel / Post, and an explicit PUBLIC marker. Writing in a sheet gives room for the text and the keyboard while costing the results layout 0px.
  • Posting a take is what applies the green comment-ring to the player's avatar in the crowd rows — the entry point and the glow indicator are two ends of one feature.

8.8 The duel — optional, opt-in

The original head-to-head "duel" is revived as an optional mode, off by default. When a player pairs with a duel partner, a compact you vs them block appears above the crowd section on the results screen — both players' picks and confidence, who matched GTO, a running set score. The default results screen (no duel) is purely the crowd database.

9. Open decisions

These were surfaced during design and need a product call before or during build.

  1. Duel enablement. Account-level pairing (pair once; the duel block then appears on every hand) vs a per-hand toggle. The mockups assume account-level. Account-level fits "the original functionality revived"; a toggle is more flexible for dueling different friends.
  2. Take editing. A posted take is currently editable indefinitely. If the crowd takes are meant to be in-the-moment reads, posted takes should lock — or show an "edited" marker — once the GTO answer has been revealed, so a take can't be quietly rewritten with hindsight.
  3. Lesson-header tag overflow. A scenario with 5+ concept tags wraps the tag row and grows the GTO summary card. Recommend a cap (show 4, "+N" overflow) for predictable card height.
  4. Matrix sizing. Fullscreen editor is the primary recommendation (§5). Confirm whether the inline overflow-x:auto fallback is needed at all.
  5. Correct-answer state. When the player matches GTO, the YOUR PICK row and the GTO LINE row are the same action. They should merge into a single green card reading YOUR PICK · ✓ GTO LINE. This combined state needs to be specced and mocked.
  6. Pending-duel state. In an async duel, the partner may not have answered the hand yet. The duel block needs a "waiting on [partner]" state.
