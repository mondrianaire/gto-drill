Mobile formatting — GTO Drill

The app is mobile-first by intent — two players, mostly on phones — but the rapid feature build pushed several surfaces past their comfortable mobile widths. This audit walks every screen at 375 px (the narrowest common iPhone) and flags every layout that's broken, cramped, or simply uncomfortable to thumb.

Telos. A two-person poker quiz on phones. Each click should land cleanly, every screen should fit without horizontal scrolling, and the tactile cadence (decide → reveal → next) should never make the user fight the layout.

This audit assumes: (a) the user is one-thumb on the phone; (b) the dominant interaction is tapping (cards, action buttons, range cells); (c) GTO prose, range grids, and the equity panel are the heaviest text-density surfaces and need the most attention.

How to read this report. The "Current" panels are <iframe>s pointing at the running dev server (localhost:8000) — open this file from the same server (http://localhost:8000/design-audit/mobile-audit.html) so they render live. The "Proposed" panels are static HTML/CSS mockups showing the redesign idea, with the relevant CSS snippet for review. Mockups intentionally don't include the full visual chrome — they show the structural change, not the final pixel polish.

Findings

H-1

Solo header overflows at narrow widths

High
Where: Solo practice view (both decide and reveal). The header row crams together "Solo practice", running stats ("Hands N · GTO accuracy X%"), Copy share link, and ← Exit solo. At 375 px the row breaks: stats wrap to two or three lines and the Exit label clips at the right edge.
The problem

The current header is a single display:flex row with four flex items (title, stats, share, exit). The stats text grows in length the longer you play ("Hands 12 · GTO accuracy 67%"), so even with flex:1 1 auto it muscles other items out. Exit clips, Copy share link sometimes wraps, and the visual hierarchy of "what's at the top of the screen" turns into a wall of text.

Why it matters

The header is the user's anchor in solo mode — it's where you read your running accuracy and where you reach to exit or share. The Exit button getting clipped is a real bug — at 375 px the visible label is "← Ex", which looks like a layout error to a casual user.

The two-row structure removes the four-way fight for horizontal space and uses the stats line as the natural anchor under the title. Touch targets stay ≥36 px for the icon buttons.

H-2

Replay table seats overlap at 375 px

High
Where: The replay table mounts inside every scenario. The 6-max ring is positioned with absolute pixel offsets (.rslot-1 { left: 4px }, etc.) and 80-px-wide seats. At 720-px container width those seat boxes are well separated; at 375 px container width (343 after padding) the three bottom seats and the three top seats get crammed against each other.
The problem

Quick math: the table is 343 px wide at 375 px viewport. Three bottom seats × 80 px each = 240 px just for seat boxes, plus 4 px padding on each outside seat = 248 px. The center seat is positioned at left: 50%; transform: translateX(-50%) with another 80 px = 320 px taken up. That leaves 23 px total for the gaps between the three seats — they visually overlap, and the hero's bet bubble overlaps the next seat's stack number.

Why it matters

The table is the centerpiece of every scenario. Seat overlap obscures position labels, stack sizes, and bet sizes — exactly the data the user reads to decide. It also makes the table look broken, undermining confidence in the rest of the UI.

Proposed fix

Two CSS changes for narrow widths: (1) shrink seat width from 80 → 64 px, (2) switch hard-coded 4 px edge offsets to percentage (left:2%). The center seat still uses translateX(-50%). Math: 64 + 64 + 64 + (8 + 8 + 14 + 14) gap = 236 px total, leaving 107 px of breathing room on a 343-px container. Seats no longer overlap, position labels and stack sizes stay readable.

Alternative (more invasive): at <380 px, switch the layout from an oval ring to a stacked list — hero seat at the top, board in the middle, opponents folded out below as a horizontal strip. Less "poker table"-like but uses screen real estate efficiently. Worth considering as part of the bigger table rework (task #14).

H-3

13×13 range grid cells fall below touch-target minimum

High
Where: The villain-range picker (the 13×13 hand grid). Used inside the per-scenario equity panel AND the standalone calculator.
The problem

At 375 px, the 13×13 grid lays out cells at roughly 22 × 18 px each. That's well below the iOS Human Interface Guidelines minimum of 44 × 44 px and Material Design's 48 × 48 px. Mishit rate on small touch targets goes up exponentially as the target shrinks — a thumb landing on the intended cell becomes a coin flip when cells are this size, especially on the diagonal pair cells.

Why it matters

Picking ranges is the highest-frequency interaction on the equity surface. If every cell tap is a 50/50 to hit the right cell, the feature is effectively unusable on phones. Right now this would push the user to use presets (Any two / Pairs / Broadways) — which is fine for those three, but the whole point of the 13×13 grid is fine-grained custom ranges.

Proposed fix

Wrap the grid in a horizontally-scrollable container at <480 px and use fixed-pixel cells (36 × 32 px). The grid takes ~507 px of inner width; the user swipes left/right to reach the off-screen columns. A subtle right-edge fade hints at the overflow.

Alternative (recommended for follow-up): add a "compact picker" mode for mobile — pick a row (A, K, Q, …) and the column expands to a single horizontally-scrollable strip of cells for that rank. Two-tap workflow but every tap is on a generous 44 × 44 button. More involved; could fold into the broader table-rework (#14) or design-system pass (#15).

M-1

4×13 hero/board card grid borderline touch target

Med
Where: The hero-hand picker (for the 2 null-hero scenarios) and the standalone equity calculator (both hero and board pickers share the same grid).
The problem

At 375 px, the 4×13 card grid lays out cells at roughly 24 × 30 px. Better than the 13×13 grid but still below the iOS/Material minimums. Each cell contains a mini playing-card glyph, so the visual is recognizable, but the tap target is small enough that adjacent-cell mishits will happen.

Why it matters

Picking hero cards is a quick 2-tap interaction; picking board cards adds up to 5 more. If even 1 in 8 taps lands on the wrong card, the user spends as much time correcting as picking — a frustrating rhythm.

Proposed fix

Same pattern as H-3: at <480 px, wrap the grid in a horizontally-scrollable container and bump cells to 36 × 44 px (full touch-target compliance). The grid stays 4 rows tall but extends ~507 px wide; users swipe between A–8 and 7–2.

An even better alternative would be to combine the hero/board picker with a "recent picks" row — most users will pick cards from a small subset (the cards mentioned in the prose). Surfacing those at the top would mean 1-tap selection for the common case. Worth exploring during #15.

M-2

Sign-in buttons wrap awkwardly on the gate screen

Med
Where: Sign-in screen (the gate before signed-in routes). The "Practice solo" and "Equity calculator" buttons.
The problem

Current labels: "🃏 Practice solo (no sign-in, no opponent)" and "🧮 Equity calculator (hole cards + board + range)". Both wrap to 2 lines at 375 px and the parentheticals dominate the button visually. The user has to read past the verbose qualifier to grasp the action.

Why it matters

Buttons should announce their action, not their disclaimers. Parentheticals belong in subtext under the button, or in tooltip-style microcopy, not stuffed inside the click target.

Restructure the button labels: bold action line + muted single-line subtext. Action reads at a glance; subtext explains. Keeps the dashed-border / muted visual style.

L-1

"GTO Drill" appears twice on the sign-in screen

Low
Where: Sign-in screen. The header at the top of the page already says "GTO Drill v…", and then the sign-in card has a huge <h1 class="appname">GTO Drill</h1> directly under it.
The problem

Visual stutter. The user sees the app name twice within ~80 px of vertical space. The landing screen for signed-in users hides the in-page appname via .landing h1.appname { display:none; } but the sign-in screen doesn't apply the same rule.

Why it matters

Minor — it's not broken, just not polished. Removing the duplicate also reclaims ~60 px of vertical space that could go to the tagline or push the buttons further up the fold.

Proposed fix

Extend the existing rule to cover sign-in too:

.landing h1.appname,
.signin h1.appname { display: none; }

Implementation priority

Suggested order — fastest wins first, then the structural changes. All six should ship as a single mobile-fixes PR after you sign off on these recommendations.

  1. L-1, M-2 — sign-in screen cleanup (trivial CSS + label restructure). ~30 min. Pure polish; quick to ship and review.
  2. H-1 — solo header restructure (two-row layout, icon-only Share/Exit at <480 px). ~45 min. Fixes a real clipping bug.
  3. H-2 — replay table seat compaction at <480 px (narrower seats, percentage offsets). ~1 hr. CSS-only minimum-viable fix; the deeper layout reconsideration is part of #14.
  4. H-3, M-1 — horizontally-scrollable grid containers for the 13×13 villain picker and the 4×13 hero/board picker. ~1.5 hr. Fixed-pixel cells + overflow-x scroll; subtle edge fade for affordance.

Out of scope

Things this audit deliberately does NOT propose, because they belong elsewhere:

  • Visual redesign of the replay table. The seat-overlap fix is structural-only. A full visual rework (felt texture, chip animations, hero/villain branding) lives in #14.
  • Unified text voice. Bb-chip, pot-% formatting, any-suit glyph — all part of #15. The mobile audit assumes those won't ship in this round.
  • Auto-animate replay open. Part of #11, absorbed by #14.
  • The "compact picker" two-step pattern (pick a rank row → expand to a single strip) for the 13×13. Worth doing but it's a UX rethink, not a layout fix; fold into #14 or #15.
  • Multiplayer in-game header. Not audited because reaching it from this anonymous session requires a live game. Suspect it has the same kind of header-overflow issue as solo — worth a follow-up screenshot review.

Verification plan (after fixes ship)

  2. Resize to 360 × 800 (Android baseline) and re-walk — should still hold up.
